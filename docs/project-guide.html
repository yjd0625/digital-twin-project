前端模块 frontend/src/

前端基于 Vite + Three.js (v0.160.1)，所有源代码在 frontend/src/ 下。

main.js 应用入口

frontend/src/main.js
整个应用的胶水模块。它把其他所有模块串联起来：初始化场景 → 加载模型 → 启动 WebSocket → 绑定 UI → 进入渲染循环。这个文件不写具体业务逻辑，只负责"谁先谁后"。
📦 它做了哪些事（按顺序）
顺序做的事想改这里…
1创建 Three.js 场景、相机、渲染器、轨道控制器改场景背景色、灯光位置 → scene.js
2绘制左下角固定坐标轴（XYZ 轴 + 标签）坐标轴大小/位置 → 直接在 main.js 的"固定坐标轴"段修改
3定义模型配置（url / 标签 / 数量 / 位置）增加/删除/移动设备 → 改 configs 数组
4加载模型模板 → 克隆实例 → 添加到场景改加载方式 → models.js
5建立 WebSocket 连接、定义收发函数改服务器地址/端口/重连逻辑 → 直接在 main.js 的 connectWebSocket 段
6初始化 importer / UI / dataHandler / interaction改对应功能 → 各自模块文件
7加载完成后自动调整相机视角到全场景改初始相机位置 → 直接在 main.js 的 loadAllModels().then() 段
8注册全局错误捕获、窗口自适应—
9启动 requestAnimationFrame 渲染循环—

data_handler.js 数据 → 模型驱动

frontend/src/data_handler.js
后端推送 JSON 数据后，DataHandler.process() 被调用，根据设备 id 找到对应模型，修改颜色、位置、标签文本。这是"数字孪生"的核心——数据动，模型动。
📤 导出 & API
名称说明
class DataHandler构造时传入 { objects: {}, updateInfo: fn }
.process(data)入口：解析 JSON，驱动所有模型
.findModelById(id)从 this.objects[id] 查找模型
.applyStatus(model, status)状态 → 颜色（绿色运行/蓝色空闲/红色停止/橙色故障）
.applyPosition(model, pos)更新模型 x/z 坐标
.updateLabel(model, text)更新 CSS2D 标签文字
.applyTemperature(model, temp)温度 → 颜色渐变（20°C 蓝 → 60°C 绿 → 100°C 红）

后端模块 backend/src/

Python 后端（FastAPI）。连接 PlantSimulation 的 TCP Socket 接收仿真数据，经 Redis 消息总线解耦后通过 WebSocket 广播给前端；前端 / REST 指令经总线反向下发 PlantSimulation。

main.py 主入口 (FastAPI)

backend/src/main.py
FastAPI 应用（uvicorn 托管），HTTP 与 WebSocket 共用端口 8300：
① WebSocket 路径 /ws（ws://localhost:8300/ws）
② 连接 PlantSimulation TCP 127.0.0.1:30000
③ lifespan 中启动后台任务：读 Plant → bus.publish("plant/state")；并订阅 plant/state → 广播前端、订阅 plant/command → 下发 Plant
④ 路由：/ws、/health、/status、POST /command（发指令改为 bus.publish("plant/command")）

websocket_handler.py WS 连接管理

backend/src/websocket_handler.py
WebSocketHandler 类维护前端连接的 set：
· handle_client() — 新前端连接/断开
· broadcast(data) — 向所有前端发 JSON

plant_connector.py PlantSim 通信

backend/src/plant_connector.py
PlantConnector 管理到 PlantSimulation 的 TCP 持久连接：
· .connect() / .send() / .recv() / .close()
阻塞接收，在 asyncio 线程池中执行。

data_processor.py 数据解析

backend/src/data_processor.py
DataProcessor.parse(raw) 尝试 JSON 解析，失败则按逗号分隔格式兜底。

config.py 配置中心

backend/src/config.py
所有可配置项集中在这里：
· PlantSim host/port: 127.0.0.1:30000
· HTTP / WS 共用端口: 8300（避开 Docker/WSL 预留区间，WebSocket 路径 /ws）
· 消息总线: BUS_TYPE(redis)、REDIS_HOST/PORT/DB/PROTOCOL、TOPIC_PLANT_STATE、TOPIC_PLANT_COMMAND
· 日志级别
支持 os.getenv() 环境变量覆盖。

bus.py 消息总线 (Redis Pub/Sub)

backend/src/bus.py
传输无关的消息总线抽象，把 PlantSimulation 与前端解耦，为将来切 MQTT 铺路：
· MessageBus 抽象基类（publish / subscribe）
· RedisBus 实现：基于 redis.asyncio，强制 protocol=2（RESP2，兼容老版 Redis 的 HELLO 限制）
· create_bus() 工厂按 BUS_TYPE 分发，将来加 MqttBus 即可切换，业务代码不动
两个主题（MQTT 风格斜杠，切 MQTT 时名字直接复用）：
· plant/state — Plant → 前端：实时状态 JSON
· plant/command — 前端 / REST → Plant：控制指令

requirements.txt Python 依赖

backend/requirements.txt
核心依赖：fastapi、uvicorn[standard]、redis（>=4.2，提供 redis.asyncio）、websockets。标准库 asyncio / json / socket / logging 为内置。
📡 数据流（从仿真到前端） 
PlantSimulation
    │  SimTalk 脚本：socket.SendLine("machine_01,TEMP,85.2")
    ▼  TCP :30000
plant_connector.py   ← 阻塞 recv()
    │  parse() → dict
    ▼
main.py: plant_read_loop → bus.publish("plant/state")
    │
    ▼  Redis Pub/Sub（redis-twin 容器, 127.0.0.1:6379）
bus.subscribe("plant/state") → websocket_handler.broadcast()
    │  ws://localhost:8300/ws
    ▼
浏览器前端 (main.js) → dataHandler.process(data) → 3D 模型更新

反向指令（前端点击 / POST /command）：
前端 WS / POST /command → bus.publish("plant/command")
    ▼  Redis
bus.subscribe("plant/command") → plant_connector.send() → PlantSimulation

启动顺序与端口总览

本系统共 5 个需手动启动的组件，外加 Docker Desktop 作为容器运行前提。请按以下顺序启动：

顺序组件启动方式占用端口
前置Docker Desktop桌面应用（建议开机自启）—
1Redis（消息总线）Docker 容器 redis-twin6379
2InfluxDB 3 Core（时序库）Docker 容器 influxdb3-core18080
3InfluxDB3 Explorer（Web 界面）Docker 容器 influxdb3-explorer8888
4后端 (FastAPI)Python 进程 python -m src.main8300
5前端 (Vite)npm run dev5173

本机端口占用一览：6379(Redis) · 18080(InfluxDB) · 8888(Explorer) · 8300(后端) · 5173(前端) · 30000(PlantSimulation Socket)。若某端口被占用，启动前需释放或改用其他端口。

🐳 Docker 基础设施（Redis + InfluxDB + Explorer） 
# 0. 先确保 Docker Desktop 已启动（托盘图标 running），否则容器无法运行

# 1. Redis（消息总线，端口 6379）
docker run -d --name redis-twin --restart unless-stopped -p 6379:6379 redis:7-alpine

# 2. InfluxDB 3 Core（时序库，端口 18080）
docker run -d --name influxdb3-core --restart unless-stopped -p 18080:18080 \
  -v influxdb3-data:/var/lib/influxdb3/data \
  influxdb:3-core influxdb3 serve --node-id=node0 --object-store=file --data-dir=/var/lib/influxdb3/data

# 3. InfluxDB3 Explorer（Web 界面，映射 8888 → 容器 8080，仅本机可访问）
docker run -d --name influxdb3-explorer --restart unless-stopped -p 127.0.0.1:8888:8080 influxdata/influxdb3-ui:1.8.0

# 验证
docker ps                                    # 三个容器状态应为 Up
docker exec -it redis-twin redis-cli ping   # 返回 PONG 即正常
curl http://localhost:18080/health           # 返回 {"status":"ok"}
# Explorer：浏览器打开 http://localhost:8888

# 重启电脑后若未自动起，手动拉起：
docker start redis-twin influxdb3-core influxdb3-explorer

以上为官方默认镜像与端口；若你本地容器名/端口不同，请按实际 docker run 配置调整。后端当前使用 Redis Pub/Sub 做解耦中转，InfluxDB 作为时序存储按需接入。
