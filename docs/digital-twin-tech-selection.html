前端可视化层（3D 渲染与交互）

负责将数据转化为可视化的三维场景，分为 Web 端引擎与原生 / 重型引擎两大流派。

Three.js

基于 WebGL 的轻量级 3D 库，需搭配 Vue / React 构建完整前端。

Web 端引擎
优点
  • 真正的零部署，浏览器输入 URL 即用
  • 前端生态极强，与后台管理系统无缝集成
  • 热更新无感，社区极其活跃
缺点
  • 原生不支持复杂物理碰撞和流体仿真
  • 处理超大模型（>50 万面）时需手动深度优化，否则卡顿
  • 画质上限受限于浏览器
开发难度
扩展性

Babylon.js

微软开源的 Web 端重型 3D 引擎，同样基于 WebGL / WebGPU。

Web 端重型
优点
  • Web 端画质天花板，原生支持 PBR 材质、环境光遮蔽和内置物理引擎
  • 工具链完整（含 GUI 编辑器）
  • 性能优于 Three.js
缺点
  • 学习曲线陡峭，中文文档和社区资源远少于 Three.js
  • 开发周期较长
开发难度
扩展性

Unity

跨平台的独立客户端引擎，支持 PC、移动端及 VR / AR（含 C# / .NET）。

原生 / 跨平台
优点
  • 性能与画质的黄金平衡点
  • 物理引擎极为成熟（NVIDIA PhysX）
  • 可编译为 WebGL（体量较大）；C# 语言相对友好
缺点
  • 部署笨重，需下载数百 MB 安装包，更新需重新下载安装
  • 与 Web 后端及工业协议（OPC UA）对接需额外编写原生插件或网络通信层，集成成本高
开发难度
扩展性

Unreal Engine（UE5）

顶级影视级独立引擎，侧重极致渲染效果。

影视级重型
优点
  • 画质天花板，Nanite 几何体技术可处理数十亿面的超精细模型
  • Lumen 动态全局光照极逼真
缺点
  • 部署困难：通常依赖 Pixel Streaming 串流，每并发用户需占用独立昂贵 GPU 实例
  • C++ 门槛极高；对硬件要求苛刻
开发难度
扩展性
2

后端服务层（业务逻辑与 API）

负责数据处理、接口暴露和通信管理。

Python

动态脚本语言，聚焦快速开发与数据分析。

FastAPI / Django
优点
  • 胶水特性，与 AI / ML 库（Pandas、NumPy）无缝衔接
  • 异步框架（FastAPI）性能优秀，开发效率比 Java 高 3–5 倍
  • 后期数据分析拓展能力强
缺点
  • 全局解释器锁（GIL）限制 CPU 密集型并行计算
  • 弱类型在大型项目中易导致维护混乱
  • 超高并发（>5000 req/s）下表现逊于 Java
开发难度
扩展性

Java

静态编译型语言，面向大型企业级分布式系统。

Spring Boot
优点
  • 极致性能与稳定性，高并发处理能力极强
  • 强类型系统配合 IDE 让重构和大型团队协作非常稳健
  • 生态极其庞大（微服务、消息队列、事务管理）
缺点
  • 代码臃肿冗长，开发慢，迭代周期长
  • 内存占用高
  • 做数据分析或 AI 集成需要额外调用 Python 服务
开发难度
扩展性
3

数据通信与接入层（OT / IT 互联）

负责打通物理设备与 IT 系统的“神经系统”。

OPC UA

工业 4.0 标准通信协议，基于 C/S 和 Pub/Sub 模型。

工业 4.0 标准
优点
  • 自带标准信息模型，连接设备后数据结构自动映射（无需手写 JSON 解析）
  • 内置强大的加密、认证和会话管理，企业级安全开箱即用
  • 支持历史数据访问和方法调用
缺点
  • 协议体量大，SDK 配置复杂（需处理证书、防火墙端口）
  • 弱网环境下重连机制编写麻烦
  • 完全实现规范开销较大
开发难度
扩展性

MQTT

轻量级物联网消息传输协议，基于发布 / 订阅模式。

物联网轻量
优点
  • 协议极轻量，头部仅 2 字节，带宽占用极小
  • 跨网穿透能力强，适合设备上云或公网传输
  • 中间件（Broker）部署极为简单（如 EMQX）
缺点
  • 数据无结构，必须自定义 Topic 和 Payload，设备变动需频繁升级解析代码
  • 安全性依赖外挂 TLS，配置复杂
  • 传输可靠性（QoS）实现不如 OPC UA 标准统一
开发难度
扩展性
4

数据存储层（数据库选型）

负责存储设备历史数据和缓存实时状态。

InfluxDB

开源时序数据库的事实标准。

时序数据库
优点
  • 生态极成熟，与 Grafana 监控看板无缝集成
  • 类 SQL 查询语言（Flux / InfluxQL）上手快
  • 数据压缩率高
缺点
  • 开源版（OSS）不支持集群，高可用需购买商业版（InfluxDB Enterprise）
  • 单机版写入瓶颈约每秒数十万点位
开发难度
扩展性

TDengine

国产高性能分布式时序数据库。

国产分布式
优点
  • 集群版完全开源免费，写入性能是 InfluxDB 的 2 倍以上
  • 原生支持标准 SQL（JDBC / ODBC），与现有数据栈融合简单
  • 超级表机制非常适合处理工业传感器数据
缺点
  • 独特的标签 / 列模型需要学习成本
  • 周边生态（数据迁移、可视化插件）不如 InfluxDB 丰富
开发难度
扩展性

Redis

内存中的键值对缓存数据库。

缓存层
优点
  • 亚毫秒级读写，用于存储最新设备实时值（Key: 设备 ID，Value: JSON 结构）
  • 极大减轻后端查询时序库的压力
缺点
  • 数据易失（依赖断电备份策略）
  • 无法做复杂历史查询
开发难度
扩展性
5

仿真与验证层（特定软件角色）

Plant Simulation 在架构中是一个独立的离线分析工具。

Plant Simulation

面向对象的离散事件仿真工具，专注物流优化、产线产能瓶颈分析和节拍验算。

离散事件仿真
优点
  • 与物理设备完全隔离，风险极低
  • 运行速度可调（加速仿真数天产能）
  • 与 Python 和 OPC UA 均有标准接口（Socket / ODBC / OPC DA）
缺点
  • 不具备实时控制能力
  • 渲染效果简陋
  • 授权费用昂贵
开发难度
扩展性
